# /plan-feature — Create Feature Plan

Feature name: `$ARGUMENTS`

If no name provided, ask the user.

## Feature Plan Naming Convention

Feature plans use a **phase + sequence** naming scheme that maps directly to `features/roadmap.md`:

```
features/planned/P{phase}-{seq:03d}-{kebab-name}/README.md
```

Examples:
- `features/planned/P1-001-ai-maintenance-intake-mvp/README.md`
- `features/planned/P1-002-clerk-auth/README.md`
- `features/planned/P2-001-rent-reminder/README.md`

To determine the next ID:
1. Read `features/roadmap.md` to identify which phase this feature belongs to
2. Scan `features/planned/` for existing `P{phase}-*` directories
3. Use the next sequence number for that phase

## Workflow

### 1. Determine Phase & Sequence

Read `features/roadmap.md`. Ask the user which phase if unclear. Find the next sequence number for that phase by scanning `features/planned/P{phase}-*/`.

### 2. Create Ticket

Find the next ticket number in `.claude/tickets.md`. Add a `new-feature` ticket.

### 3. Create Branch

```bash
git checkout -b feature/T-NNN-<feature-name>
```

### 4. Gather Requirements

Ask the user targeted questions:
- What problem does this solve?
- Who uses it (tenants, landlords, vendors)?
- What are the inputs and outputs?
- Any constraints or preferences?

### 5. Research

- Read `intake/readme.md` for product context and MVP scope
- Read existing `features/planned/` for consistency with prior plans
- Search the codebase for related code
- Web search for relevant APIs, libraries, or patterns if needed

### 6. Write Feature Plan

Create `features/planned/P{phase}-{seq:03d}-{feature-name}/README.md`:

```markdown
# Feature: <Name>

**ID**: P{phase}-{seq:03d}
**Ticket**: T-NNN
**Phase**: {phase number} — {phase name}

## TL;DR
One-sentence summary.

## Summary
What it does and why it matters for the Liz platform.

## User Stories
- As a [landlord/tenant/vendor], I want to...

## Architecture
How it fits into the existing system. Diagrams if helpful.

## Tech Approach
Key technical decisions, libraries, APIs.

## Data Model
New schemas, modifications to intake.json format, database tables.

## Integration Points
How it connects to existing features (intake, vendor matching, tenant comms).

## Manual Testing Checklist
Concrete test cases the user can run manually to verify the feature works.

## Tasks
_Generated by `/create-feature-tasks-in-backlog <feature-name>` into the feature's `backlog/` directory_

## Open Questions
Unresolved decisions that need user input.
```

### 7. Update Tracking

- Add/update entry in `features/roadmap.md` with the feature ID (e.g., `P1-001`)
- Add decision to `plan/DECISION_LOG.md` (create if doesn't exist)
- Update ticket branch in `.claude/tickets.md`

### 8. Report

- Feature ID (e.g., P1-001)
- Ticket number
- Branch name
- Feature plan path
- Next step: `/create-feature-tasks-in-backlog <feature-name>`
