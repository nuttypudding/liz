| 2026-04-28 | Three deploy targets, one codebase for agents | Local dev = native Python with `uvicorn --reload` (fast iteration, no Docker overhead). | 2026-04-28 | Monolithic Vercel deploy for agents (initially) | All agents share one Vercel project (`liz-agents`), each mounted under a path prefix (`agents.brightstep.ai/triage`, `agents.brightstep.ai/vendor-dispatch`, etc.). Vercel doesn't bill per project, so cost wasn't the driver — but operational simplicity is. Per-agent code stays isolated in `agents/<name>/` so splitting into separate projects later is mechanical (no source changes). Trade-off: deploys are coupled, function size cap (250MB) is shared. Split when pain shows. | Active |
| 2026-04-28 | Agent ↔ Liz auth: shared secret in header | Service-to-service `X-Agent-Auth` header. One secret per agent, in env. No Clerk on agent endpoints (server-to-server only). Swap to short-lived JWTs when multi-tenant or when secrets sprawl. | Active |
| 2026-04-28 | Agents share Liz's Supabase, no schema fork | Agents use the service-role key. Writes are tagged `created_by='agent:<name>@<version>'` for audit. Migrations stay in `apps/web/supabase/migrations/`. Hard rule — agent-specific schemas would create debugging hell. | Active |
| 2026-04-28 | No streaming for v1 agent contract | Triage flow is <5s end-to-end, fits Vercel function limits cleanly. `/v1/run` only. Add `/v1/run/stream` to the standard later when an agent needs it (likely tenant-comms or any conversational flow). | Active |

| 2026-04-28 | Reconsider agent runtime: framework-per-agent, no monoculture | After GTC content highlighting LangChain/LangGraph long-running agents, revisited the "MAF for everything" decision. New rule: each agent picks the lightest framework that fits. v1 triage — stateless, vision, <5s, no tools — uses raw `openai` SDK + ~50 lines of agent logic. Future stateful agents (vendor dispatch, autonomy, comms) default to LangGraph when checkpointing/long-running state is needed. Rationale: MAF is young (Nov 2024 preview), small ecosystem, and not differentiated for stateless agents. LangGraph has the strongest long-running story but adds weight not justified for triage v1. The HTTP contract is the abstraction; framework is an implementation detail per agent. | Active (supersedes the MAF entry above) |
| 2026-04-28 | LLM access via OpenRouter, not direct provider SDKs | All agents call LLMs through OpenRouter using the `openai`-compatible Python SDK with `base_url=https://openrouter.ai/api/v1`. Model is configurable via env (`AGENT_<NAME>_MODEL`, e.g. `anthropic/claude-sonnet-4-6`). One API key, ~200 models, swap providers via env without code change. Trade-offs accepted: ~5% margin on token cost vs direct provider, ~50–100ms extra latency, one more vendor in the data path, prompt-caching behavior to verify on first deploy. Rationale: future model swaps and multi-model evals are a primary goal; portability beats marginal cost/latency at MVP scale. | Active |
| 2026-04-28 | Triage agent v1 implementation: raw OpenAI SDK + OpenRouter, no agent framework | ~50 lines: system prompt + one chat completion call (with image content when photos present) + JSON-schema-validated parse. No agent framework dependency. Default model `anthropic/claude-sonnet-4-6` via OpenRouter, switchable via `AGENT_TRIAGE_MODEL` env. Eval suite re-runs against 2–3 alternate model IDs (e.g. `openai/gpt-5`, `google/gemini-2.5-pro`) on first deploy to verify the abstraction holds. If/when triage v2 needs tools/state, revisit framework choice — likely LangGraph. | Active |
